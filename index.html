CIPHER PIPELINE
1. PARSE — tokenize + AST-like IR
2. COMPILE — IR → bytecode opcodes
3. SHUFFLE — randomize opcode values
4. STRING TABLE — extract all literals
5. XOR-256 — encrypt string table
6. POLY-WRAP — encode VM bootstrap
7. LAYER N — repeat N times
8. HEADER — embed recovery metadata
RECOVERY HEADER
Hidden as: /* VMCX:{base64} */
Contains: session key, opcode map,
string table offsets, layer count.
Invisible to casual inspection.
Only our deobfuscator reads it.
WHY IT'S HARD TO REVERSE
• Opcode values are randomized per build — no two builds share the same opcode table
• String table is encrypted with a per-session 256-bit XOR key derived from runtime entropy
• The VM dispatcher is itself re-encoded via polynomial substitution each run
• Variable/register names change every build — no stable symbol to anchor analysis
• Control flow runs through a state-machine switch with shuffled state ordering
• Multiple layers of dead code and opaque predicates inserted between every real operation
• Proxy function layer wraps all calls — static analysis cannot determine call targets
• Without the session key + opcode map, decryption is computationally infeasible
